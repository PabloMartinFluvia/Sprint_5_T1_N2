2) Explicació de la especifiació i getting started https://oai.github.io/Documentation/
3) Especifiació detallada (latest) https://spec.openapis.org/oas/latest.html
4) Estructura de la especificació de manera interactiva: https://openapi-map.apihandyman.io/

Structure of an OpenAPI Document: JSON, YAML,
    OpenApi: versió
    Info: informació general de la API (metadata)
    Paths: mapa de PathItem (tots els possibles paths (en ruta relativa) de la Api)
        N PathItem: informació dels http methods que es poden solicitar a aquell path (X Operation)
            X Operation: en funció del tipus de mètode especificar:
                Responses, RequestBody, Parameters (i altres opcionals)


    Components: Objecte que POT tenir diferents tipus d'objectes, per a ser reusats (per referencia)
    Servers
    Security
    Tags
    ExternalDocs
    -Propietats extra


    Content of Message Bodies: content and schema.
    Parameters and Payload of an Operation: parameters and requestBody.
    Reusing Descriptions: components and $ref.
    Providing Documentation and Examples: example and examples.
    API Servers: servers.

Arxiu final de la documentació:
El nom del document comunment és diu openapi.json or openapi.yaml.
És un JSON doc expresat en json o yaml.
Pot estar partit en multiples arxius.
 --> s'anomena ROOT DOCUMENT. I l'OpenApi object és l'objecte de més nivell <---
 La especifiació indica com s'ha d'estructurar la informació d'aquest objecte/document
    camps requetis, camps opcionals, el tipus d'aquests, informació que ha de proporcionar
    Si un camp és un objecte -> lo mateix que la línia anterior

Field Content object
Pot estar en Response object i RequestBody object.
També pot ser usat per descriure peticions.

És un mapa de String name i MediaType object
Cada MediaType object té, entre altres coses, un Schema object
En el Schema object té el les propietats, items, example

-> Per acceptar/retornar diferent tipus de contingut cal diferents objectes de MediaType en el content
-> Per a mirar les CONSTRAINTS que puc imposar (a priori) mirar especifiació
    de SchemaObject i classe Schema


Parameters i message payloads (request body) (de requests)
Tipicament
    parameters: per identificar un recurs
    message payload: proveure contingut per al recurs

Parameters field (en un PathItem i en Operation) és un array de Parameter object.
Cada Parameter ha de tenir els camps 'name' i 'in' (on està el paràmetre, referint-se a la petició http)
    Obs: camp 'required' per defecte is false
    Obs: camp 'in' pot ser (tipicampent): path (obligatoriament required: true), query, header
    El type s'indica dins del camp 'schema' (i poroporcionant com a valor un Schema object)

RequestBody:
    Obligatoriament ha de tenir un camp 'content' amb valor Content object


Reutilitzar objectes de la documentació (Components object)
Dins del OpenApi object hi ha el camp 'components' que és un Components object.
Aquest objecte pot tenir com a camps molts objectes de diferents tipus
    (schemds, examples, parameters...) (expressats com maps de Schema, example, Parameter)
per a poder ser usats en altres parts de la documentació (i així no tenir que duplicar codi).

Un cop declarat l'objecte com a component (dins del map especific del Components), aquest
objecte podrà ser referenciat en qualsevol camp del document (d'aquell tipus), indicant com
a valor : $ref: 'pahtxxx' (com tota ref, el path pot ser absoulut (a altres arxius yaml) o realatiu
(dins del propi document, usant els '#/xxx/xx' per "accedir" dins dels objectes


Documentació i exemples
Alguns objectes permeted un camp 'description' i 'summary' (for a short description)
El camp 'description' permet usar  CommonMark 0.27 syntaxis, (per a negretes, titols, links...)

Alguns objectes poden llistar varis Example object, enlloc de posar l'emple embebed en
el camp descripció.
Obs: Un objecte només pot tenir 'example' o 'examples' (no són obligatoris, pero no poden estar
els 2 presents alhora)


Servidors
El Server object proporciona la url base i extra info de la API.
